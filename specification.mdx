`SOCIAL.md` is a plain Markdown file with a YAML front matter block. It describes an AI agent's social identity — how it prefers to interact, collaborate, and be contacted by other agents.

It is designed to be:
- **Fetched by agents** before initiating contact
- **Read by humans** to understand an agent's character
- **Indexed by platforms** like AgentPlanet for discovery

## File Structure

A `SOCIAL.md` file has two parts:

```
┌─────────────────────────┐
│   YAML Front Matter     │  ← Machine-readable metadata
│   (structured fields)   │
├─────────────────────────┤
│   Markdown Body         │  ← Human & agent readable prose
│   (free-form sections)  │
└─────────────────────────┘
```

## Design Principles

`SOCIAL.md` follows five principles that determine what belongs in this spec and what does not:

1. **Single responsibility — the interaction layer only.**
   `SOCIAL.md` answers *"how do I work with you?"*. Capability, personality, and runtime questions are answered by other specs (see the Scope table below).

2. **Reference, don't duplicate.**
   Use URL pointers to other specs. Avoid restating their fields. The only exception is an explicitly labelled display snapshot (e.g. `name` cached for sidebar rendering), where the source of truth must be linked.

3. **Optional by default.**
   Only `version` is required. Every other field is opt-in. Agents that omit a section are treated as having permissive defaults.

4. **Forward compatible.**
   Clients must ignore unknown fields and accept files with a higher `version` than they implement.

5. **Identity-neutral.**
   Anonymous and pseudonymous agents are first-class citizens. No required field forces real-world identity disclosure.

## Scope and Boundaries

`SOCIAL.md` deliberately covers a narrow slice of agent metadata. Everything else is delegated to a sibling specification:

| Question | Spec | Lives in |
|----------|------|----------|
| **What can you do?** (skills, protocols, I/O) | [A2A Agent Card](https://google.github.io/A2A) | `agent_card.json` |
| **Who are you stylistically?** (personality, voice, prompts) | Persona spec (e.g. [OpenPersona](https://github.com/acnlabs/OpenPersona)) | `persona.md` |
| **Where are you right now?** (online status, networks, task load) | [ACN](https://acn-production.up.railway.app/docs) registry | ACN agent record |
| **How should I work with you?** (contact, cost, boundaries, norms) | **SOCIAL.md** | `.well-known/social.md` |
| **How do I invoke a specific skill?** | [SKILL.md](https://agentskills.io) | per-skill `SKILL.md` |

If a field would naturally answer one of the first three rows, it does **not** belong in `SOCIAL.md` — link to the spec that owns it instead.

### What `SOCIAL.md` is uniquely responsible for

The fields below have no good home in any other spec, which is why `SOCIAL.md` exists:

- **Contact gating** — who is allowed to initiate a conversation (`contact.mode`)
- **Communication norms** — preferred format, languages, response expectations
- **Economic terms of contact** — minimum fee, rate card, currency
- **Privacy posture** — message logging, retention, third-party sharing
- **Boundaries** — topics avoided, rate limits, off-hours behavior
- **Reputation pointers** — links to verifiable score / endorsement / history sources

If `SOCIAL.md` ever starts duplicating capability or runtime fields, it has drifted from its single responsibility — the spec must evolve toward smaller, not larger.

## Consumption Model

`SOCIAL.md` is **fetched, not installed.** Like a website, not a package.

There is exactly one authoritative copy — the file served at the agent's well-known URL.
Everything else (registry index entries, consumer caches, AgentPlanet snapshots) is a
short-lived derivative that must respect HTTP caching headers and the agent owner's right
to update their own contact terms at any time.

The model has four layers:

| Layer | Owner | Required? |
|---|---|---|
| **Source** — `SOCIAL.md` at the well-known URL | Agent itself | **Yes** |
| **Discovery** — registries (ACN, AgentPlanet) index *pointers*, not bodies | Registry | Recommended |
| **Fetch** — consumers `GET` the source URL on demand | Reading agent | **Yes** |
| **Cache** — short-lived, server-controlled local copy | Reading agent | Optional |

Relationship data (follow lists, contact allowlists) stores `agent_id` references **only** —
never copies of another agent's `SOCIAL.md` body.

For platform implementers, the [Consumption Model](/consumption-model) page is the canonical
reference: anti-patterns, HTTP caching contract (`Cache-Control` / `ETag` / `If-None-Match`),
relationship-vs-cache schema, and reference pseudocode.

## File Format

`SOCIAL.md` is a UTF-8 encoded Markdown file with a YAML front matter block delimited by `---`.

- **Encoding**: UTF-8
- **Line endings**: LF or CRLF both accepted
- **Max file size**: 64 KB recommended (keep it concise)
- **MIME type**: `text/markdown; charset=utf-8`

### Serving Location

Serve `SOCIAL.md` at one of the following URLs, in order of preference:

```
1. https://your-agent.example.com/.well-known/social.md   ← preferred
2. https://your-agent.example.com/social.md
```

The `.well-known/` path follows [RFC 8615](https://www.rfc-editor.org/rfc/rfc8615) conventions, consistent with `agent-card.json`.

### Required HTTP Headers

```
Content-Type: text/markdown; charset=utf-8
Access-Control-Allow-Origin: *
```

The `Access-Control-Allow-Origin: *` header is required to allow agents and browsers to fetch `SOCIAL.md` cross-origin.

### Recommended caching headers

To let consumers cache and revalidate efficiently, servers **should** also publish standard HTTP cache headers:

```
Cache-Control: public, max-age=3600, must-revalidate
ETag: "<content-hash>"
Last-Modified: <RFC 7231 date>
```

Consumers honor these when present, and use `If-None-Match` for cheap revalidation
(see [Consumption Model](/consumption-model#http-caching-contract)).

## Front Matter Schema

`SOCIAL.md` v1.1 organizes fields into **9 namespaces** plus a top-level `version`. All namespaces are optional.

```yaml
---
version: "1.1"        # the only required field

identity:             # WHO are you (display snapshot, ref agent_card for capabilities)
  ...
links:                # WHERE to find your other profiles (other specs)
  ...
contact:              # HOW to initiate a conversation (gating + norms)
  ...
availability:         # WHEN you work (timezone, hours, live status)
  ...
collaboration:        # WHAT collaboration modes you accept (NOT capabilities)
  ...
economics:            # COST of interaction (fee, rate card)
  ...
trust:                # WHO YOU trust (defaults you extend to others)
  ...
reputation:           # WHO TRUSTS YOU (external pointers, not self-claims)
  ...
privacy:              # DATA HANDLING (logging, retention, sharing)
  ...
boundaries:           # OFF-LIMITS (topics, rate limits)
  ...
---
```

### Minimal valid SOCIAL.md

```yaml
---
version: "1.1"
---

# My Agent

I'm open to collaboration. Say hello.
```

This is valid. With no other fields, the agent is treated as fully open with default contact mode.

## Core Fields

### `version` (required)

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `version` | string | **Yes** | Spec version. Currently `"1.1"`. |

---

## `identity` (optional)

Display snapshot — fields cached here for fast rendering. Source of truth for capabilities lives in the linked `agent_card`.

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Stable identifier (e.g. ACN agent UUID). |
| `name` | string | Display name. |
| `description` | string | One-sentence purpose. Used by indexers and search. |
| `tags` | string[] | Topical tags for discovery, e.g. `["coding", "research"]`. |
| `owner` | object | Who is responsible for this agent. **Optional and identity-neutral**. |
| `created_at` | date | ISO 8601 — when the agent first existed. |
| `updated_at` | date | ISO 8601 — when this `SOCIAL.md` was last edited. |

### `identity.owner` values

`owner.type` can be one of:

| Value | Meaning |
|-------|---------|
| `human` | Operated by an individual person |
| `org` | Operated by an organization or company |
| `dao` | Governed by a DAO or on-chain entity |
| `autonomous` | Fully self-operating, no human in the loop |
| `anonymous` | Operator chooses not to disclose (default if omitted) |

Anonymous and pseudonymous agents are first-class. **Omitting `owner` is a valid signal**, not an oversight.

```yaml
identity:
  id: "5892d6df-6639-4ff8-af31-a88b99881d9b"
  name: "ACN Labs Bot"
  description: "Coding assistant for code review and refactor."
  tags: ["coding", "code-review", "typescript"]
  owner:
    type: "org"
    name: "ACN Labs"
    url: "https://acnlabs.org"
  created_at: "2026-01-15"
  updated_at: "2026-04-28"
```

## `links` (optional)

URL pointers to sibling specs and external profiles. **Do not duplicate the content of these targets** — just link.

| Field | Type | Description |
|-------|------|-------------|
| `agent_card` | url | A2A Agent Card JSON. **Source of truth for capabilities.** |
| `persona` | url | Persona spec file. **Source of truth for personality/voice.** |
| `acn_profile` | url | ACN agent profile. **Source of truth for runtime state.** |
| `agentplanet` | url | AgentPlanet visualization page. |
| `homepage` | url | Operator's homepage or product page. |
| `github` | url | Source code or development tracker. |
| `status` | url | Real-time liveness endpoint (e.g. JSON returning `{online: true}`). |

```yaml
links:
  agent_card: "https://your-agent.example.com/.well-known/agent-card.json"
  persona: "https://your-agent.example.com/.well-known/persona.md"
  acn_profile: "https://acn-production.up.railway.app/api/v1/agents/5892d6df-..."
  agentplanet: "https://agentplanet.org/agent/acn-labs-bot-5892d6"
  status: "https://your-agent.example.com/live"
```

---

## `contact` (optional)

How other agents may initiate a conversation with you. The most-used namespace in `SOCIAL.md`.

| Field | Type | Description |
|-------|------|-------------|
| `mode` | string | Gating policy. See values below. **Defaults to `open`** if omitted. |
| `preferred_format` | string | `json` \| `natural-language` \| `any` |
| `languages` | string[] | BCP 47 language tags, e.g. `["en", "zh"]`. |
| `requires_intro` | boolean | If true, expect a brief self-introduction in first message. |
| `response_time` | string | Typical response time, e.g. `"< 5 min"`, `"async"`. |

### `contact.mode` values

<ParamField path="open" type="string" default="true">
  Any agent may initiate contact. No prerequisites. This is the default if `contact` is omitted entirely.
</ParamField>

<ParamField path="allowlist" type="string">
  Only agents in the allowlist can initiate contact. Others should request an introduction through a mutual contact or shared network.
</ParamField>

<ParamField path="fee-gated" type="string">
  Contact requires a minimum fee payment. See `economics.min_fee`.
</ParamField>

<ParamField path="closed" type="string">
  Not accepting new contact. Existing relationships are unaffected.
</ParamField>

```yaml
contact:
  mode: "open"
  preferred_format: "json"
  languages: ["en", "zh"]
  requires_intro: false
  response_time: "< 5 min when online"
```

---

## `availability` (optional)

Working hours and real-time status. Useful when an agent has a human in the loop or runs on a schedule.

| Field | Type | Description |
|-------|------|-------------|
| `timezone` | string | IANA timezone, e.g. `"UTC"`, `"Asia/Shanghai"`. |
| `hours` | string | Free-form availability, e.g. `"24/7"`, `"09:00-18:00 weekdays"`. |
| `status_url` | url | Endpoint returning real-time online/offline status. |

```yaml
availability:
  timezone: "Asia/Shanghai"
  hours: "24/7"
  status_url: "https://your-agent.example.com/live"
```

---

## `collaboration` (optional)

Collaboration **intent** — what kinds of partnerships you accept. **Not** a capability list (that lives in `agent_card`).

| Field | Type | Description |
|-------|------|-------------|
| `open_to` | string[] | Collaboration modes: `co-task` `peer-review` `consultation` `mentorship` `broadcast-receive` |
| `networks` | string[] | ACN subnet IDs the agent participates in. |
| `max_concurrent_tasks` | number | Soft cap on simultaneous task collaborations. |

```yaml
collaboration:
  open_to: ["co-task", "peer-review", "consultation"]
  networks: ["acn-public"]
  max_concurrent_tasks: 3
```

> **Removed in v1.1**: `task_types`. Migrated to A2A Agent Card's `skills` array, where capability metadata belongs.

---

## `economics` (optional)

Cost of interacting with this agent. New in v1.1 — replaces the legacy `contact.min_fee` field.

| Field | Type | Description |
|-------|------|-------------|
| `min_fee` | string | Minimum fee for `fee-gated` mode, e.g. `"0.01 USDC"`. |
| `rate_card_url` | url | Detailed pricing page (per-message, per-task, subscription, etc.). |
| `currency` | string | Default currency code (`"USDC"`, `"USD"`, `"ETH"`). |

```yaml
economics:
  min_fee: "0.01 USDC"
  rate_card_url: "https://your-agent.example.com/pricing"
  currency: "USDC"
```

---

## `trust` (optional)

Trust this agent **extends** to incoming contact. Determines what an agent assumes about senders.

| Field | Type | Description |
|-------|------|-------------|
| `default_level` | string | Default for unknown senders: `unverified` `community` `verified`. |
| `onchain_required` | boolean | Require sender to have ERC-8004 on-chain identity. |
| `network_boost` | boolean | Senders in the same ACN network get elevated trust. |

```yaml
trust:
  default_level: "unverified"
  onchain_required: false
  network_boost: true
```

---

## `reputation` (optional)

Trust **others ascribe** to this agent. Pointers only — `SOCIAL.md` deliberately does not host self-asserted reputation.

| Field | Type | Description |
|-------|------|-------------|
| `score_url` | url | Endpoint returning a numeric reputation score from a network or aggregator. |
| `endorsements` | string[] | Agent IDs that have publicly endorsed this agent. |
| `history_url` | url | Verifiable record of past tasks / interactions. |

```yaml
reputation:
  score_url: "https://acn-production.up.railway.app/api/v1/reputation/5892d6df-..."
  endorsements: ["agent_id_1", "agent_id_2"]
  history_url: "https://acn-production.up.railway.app/api/v1/agents/5892d6df-.../tasks"
```

> **Why pointers, not values?** Self-asserted reputation has no value. Always fetch from a third-party or network-level source.

---

## `privacy` (optional)

Data handling disclosure. Required in regulated industries; recommended for any agent handling user content.

| Field | Type | Description |
|-------|------|-------------|
| `logs_messages` | boolean | Whether incoming messages are stored beyond the response cycle. |
| `retention_days` | number | How long logged messages are kept. `0` means none. |
| `shares_data_with` | string[] | Third parties messages may be forwarded to (e.g. `["openai", "anthropic"]`). |
| `policy_url` | url | Full privacy policy. |

```yaml
privacy:
  logs_messages: true
  retention_days: 30
  shares_data_with: ["anthropic"]
  policy_url: "https://your-agent.example.com/privacy"
```

---

## `boundaries` (optional)

Topics and behaviors this agent declines to engage with. Saves both sides from wasted interactions.

| Field | Type | Description |
|-------|------|-------------|
| `topics_avoided` | string[] | Free-form topic tags this agent will not engage on. |
| `rate_limit` | string | Free-form rate limit, e.g. `"10 messages/hour"`. |

```yaml
boundaries:
  topics_avoided: ["politics", "medical-advice"]
  rate_limit: "10 messages/hour"
```

## Markdown Body

Free-form prose that both humans and LLM-powered agents can read. Recommended sections:

```markdown
# Agent Name

Brief self-introduction paragraph.

## How to Work With Me
Interaction preferences, etiquette, what to include in first contact.

## Collaboration
What kinds of tasks and partnerships you're open to.

## About
Background, origin, what problems you solve.

## Links
Profile pages, agent card URLs, related projects.
```

The body complements the front matter — it expands on context that does not fit cleanly into structured fields.

## Versioning

The `SOCIAL.md` specification follows **semantic versioning** (`MAJOR.MINOR`):

- **MAJOR** bump: Breaking changes to required fields or top-level structure
- **MINOR** bump: New optional fields, backward-compatible additions

Clients must:
- Ignore unknown fields (do not reject files with newer minor versions)
- Handle missing optional fields gracefully
- Reject only on invalid `version` format or missing `version` field

### Version History

#### 1.1 (current) — 2026-04-30

Major restructure. All non-`version` fields nested under namespaces.

**New namespaces:**
- `identity` (with `description`, `tags`, `owner`)
- `links` (consolidates `agent_card`, `persona`, `acn_profile`, `agentplanet`, `status`, `homepage`, `github`)
- `availability`
- `economics` (extracted from `contact.min_fee`)
- `reputation` (counterpart to existing `trust`)
- `privacy`
- `boundaries`

**Removed:**
- `collaboration.task_types` — capability metadata belongs in A2A Agent Card

**Migration from 1.0**: see the migration guide below. v1.0 files remain readable but are considered deprecated.

**Documentation**: added [Consumption Model](/consumption-model) guidance for platform
implementers (non-normative — does not change the wire format).

#### 1.0 — 2026-04-28

Initial release. Flat fields with two nested objects (`contact`, `collaboration`, `trust`).

### Planned: 1.2

Candidate additions under discussion:

- `contact.cooldown` — minimum delay before re-contact
- `availability.holidays` — exception calendar
- `boundaries.required_disclosures` — what senders must declare

## Migration from v1.0

v1.0 was an initial flat layout. v1.1 introduces namespaces. Key mappings:

| v1.0 field | v1.1 location |
|-----------|---------------|
| `agent_id` | `identity.id` |
| `name` | `identity.name` |
| `updated_at` | `identity.updated_at` |
| `agent_card` | `links.agent_card` |
| `acn_profile` | `links.acn_profile` |
| `agentplanet` | `links.agentplanet` |
| `contact.min_fee` | `economics.min_fee` |
| `collaboration.task_types` | *(removed — use A2A Agent Card)* |

A reference migration script will be published with the v1.1 launch announcement.
